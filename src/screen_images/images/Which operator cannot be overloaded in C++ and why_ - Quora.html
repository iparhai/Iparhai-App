These operators cannot be overloaded
1. ?: (conditional)

2. . (member selection)

3. .* (member selection with pointer-to-member)

4. :: (scope resolution)

5. sizeof (object size information)

6. typeid (object type information)

Overloading of .(dot) operator can lead to questions about whether an operation is meant for the object overloading of .(dot) or an object is referred by .(dot).

sizeof operator could not be given a new and different meaning without violating basic language rules.

There is no specific reason for not overloading ?: operator as there is no need to introduce the special case of overloading a ternary operator.

You can’t overload ::, .*, ., or ?:. This is mainly because their semantics are too different from simple function-call semantics.

Take the ternary operator (?:), for instance. Here, you’ve got three arguments, but only one of the second and third argument will ever get evaluated, depending on the value of the first argument. So the expression a ? b++ : c++ can’t be represented by some function call expression operator?:(a, b++, c++), since the function call demands that both b and c be incremented, while the ternary expression would only increment one of them.

Generally these operators can't be overloaded because overloading them could and probably would cause serious program errors or it is syntactically not possible.

For instance, the sizeof operator returns the size of the object or type passed as an operand.

It is evaluated by the compiler not at runtime so you can not overload it with your own runtime code.

It is syntactically not possible to do so.

Even if it was a pointer, the arithmetic relies on the correct value being returned by this operator since the compiler already knows how to calculate the correct value all overloading would do would be to allow you to calculate an incorrect value, something that would almost certainly lead to the program not working correctly.

Scope resolution and member access operators work on names rather than values.

C++ has no syntax for writing code that works on names rather than values so syntactically these operators can not be overridden.

Again what useful purpose would overloading the conditional operator produce? I can think of none.

Also, All operators that can be overloaded must have at least one argument that is a user-defined type. That is, operator overloading can apply only to your own classes.

That means you can't overload operator+ for int or double.

That means you can'toverload the scope resolution operator since it has no arguments.

That means you can't invent new operators.

As for # and ##, my opinion is that the concept of overloading them just doesn't make any sense, because the preprocessor doesn't know about types, so it has nothing to dispatch on. I further believe that the ability to redefine them would not be useful.

You also can't overload typeid, it's true, but that's not referred to as an operator anyway. No idea why. Maybe it is an operator, and they just don't call it that. There's no precise definition of what an operator is. As for why it can't be overloaded, I don't have a reason from an official source. I suspect that nobody ever seriously wanted it to be overloadable. If I had to give a reason for not making it overloadable, though, I would say that the implementation of the type std::type_info is supposed to be implementation-defined, and so there would be no portable way to overload typeid.

C++11 adds alignof and noexcept. Neither is overloadable. alignof is also not referred to as an operator, but is said to have an operand, so I guess it is an operator when you read between the lines. alignof is presumably not overloadable for the same reason as sizeof: the implementation needs the actual alignment in order to follow basic language rules. [1] As for noexcept, which is referred to as an operator, there is clearly no useful way to write an overloaded implementation as a function, since the value of noexcept(expr) depends on the entire AST and not just the type and value of expr.

[1] Stroustrup didn't address the possibility that incrementing pointers and other built-in features would still work normally but you could overload sizeof for a type T so that whenever the programmer typed sizeof(T) or sizeof(x) where x is of type T it would give the overloaded value. The problem with this is probably that it breaks the standard library, which needs to know the sizes of objects (consider for example allocators). Same goes for alignof.

Along with the operators that can't be overloaded, it's worth noting a few others that almost never should. The most obvious are the logical-and (&&) and logical-or (||) and comma operators.

The "native" versions of all these have the property that they evaluate their left operand, then their right operand and in the case of the logical operators, the right operand is only evaluated if it's needed to determine the overall result. Overloading them loses those properties--an overloaded version always evaluates both operands, and in an unpredictable order.
